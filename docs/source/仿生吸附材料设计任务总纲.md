## 一、课题背景与目标

仿生吸附材料设计课题隶属 ADRMATS 多智能体系统框架下的材料发现方向。由于智能体系统完成时间不确定，决定先行推进：让周佳奇基于现有仿生库或直接调用 GPT 等 SOTA 大模型辅助设计，不等智能体。

最终目标不是"做出一个仿生吸附剂"，而是设计一种能合成的仿生吸附材料，且使本项目此前所有的思路（为什么要仿生、多维度对应、痛点-仿生因果链、引言逻辑闭环）贯通起来，形成可发表的工作。

---

## 二、核心立论：为什么要仿生

当前团队存在一个思维陷阱：一直在想"如何设计仿生吸附材料"，没有退一步问"为什么要仿生"。这直接影响技术路线的合理性。写论文时，引言开头必须回答清楚切入的实际难题或难点，不能为仿生而仿生。

仿生的独特价值，必须落在**传统合成方法做不到的事情**上，至少是其中之一：

- 结构层级性（多尺度有序孔道，如木质部导管层级）
- 动态响应/可逆切换（环境刺激下构象变化）
- 多组分协同自组装（像蛋白质折叠那样精确组装）

讲不清"仿生带来传统方法做不到的能力"，仿生就只是包装，审稿人一眼看穿。

## 三、技术路线的层次结构

此前讨论过三种设计思路，三者的关系不是三选一，而是回答不同层次的问题：

| 层次 | 问题 | 对应思路 |
|------|------|----------|
| Why（立论层） | 为什么需要仿生 | 思路二：解决工程共性难题 |
| How（方法层） | 如何仿生 | 思路一：机制匹配寻找生物原型 |
| Case（验证层） | 针对具体污染物的可行性 | 思路三：针对特定污染物找特定原型（可选） |

**思路一（自下而上）**：先分析污染物性质，匹配它如何被吸附去除（如氨基对羧基、静电吸引等基团匹配）；通过这些化学/物理机制，去自然界寻找具备相同机制的生物原型；研究生物原型如何实现该机制，模仿其基团或结构，转化为设计思路。

**思路二（自上而下）**：设定要解决吸附中的共性工程难题（如孔道问题、吸附剂亲疏水性、阵列结构等），不针对某一种特定污染物，而是解决共性工程应用瓶颈。逻辑链最硬：传统方法解决不了 → 生物界进化已解决 → 借鉴。切入点多，可成系列工作。

**思路三**：针对某种特定污染物，寻找能特定吸附它的生物原型进行仿生设计。逻辑牵强，"因为难去除，所以自然界有现成方案"这一步不成立。倾向放弃，或仅作个案补充。

**主干路线**：以思路二立论（回答 why），思路一方法落地（回答 how，且必须走到结构/动态层面，不能停在官能团），思路三仅作可选验证。

## 四、真正仿生的判据：多维度对应

四个维度对应上，才能说明是真正仿生；否则只是包装成仿生，审稿人一眼看穿：

| 层次 | 维度 | 对应的痛点 |
|------|------|-----------|
| 静态结构 | 分级孔道、亲疏水 | "污染物为什么难去除"（本征吸附能力不足） |
| 动态行为 | 刺激响应、再生 | "为什么难以设计吸附剂"（工程瓶颈） |

这个分组把两类痛点切开，引言结构有了骨架：痛点1（难去除）对应仿生静态结构能力，痛点2（难设计吸附剂）对应仿生动态行为能力。

**仿生保真度**：明确说做的不是单维度仿生（只抄结构或只抄机制），而是多维度协同仿生（结构 + 机制 + 动态行为）。明确列出仿了哪几个维度、每个维度达到什么保真度，本身就是论文的 contribution，也是防"为仿生而仿生"质疑的盾牌。

**硬对应与软对应**：四个维度全对应是理想标准。周佳奇研二，实验体量有限，做到四个全对应可能不现实。建议分硬对应（论文必须证明，建议2个）和软对应（顺带提，可不深入），把硬对应做实比四个都浅做更值。

## 五、引言写作框架：痛点-仿生因果链

设计阶段即须设想清楚，写引言时痛点必须正好对应后面仿生所能解决的点：

1. 痛点1：这个污染物在目标场景下为什么这么难去除？
2. 痛点2：为什么难以设计对应的吸附剂？
3. 上述痛点 ←→ 仿生方案能解决的能力（分级孔道/亲疏水/刺激响应/再生），一一对应

**关键原则**：对应不是并列清单（痛点A对应能力A，痛点B对应能力B），而是一条贯通的因果链：

> 污染物难去除的根因 → 该根因恰是传统合成解决不了的 → 某生物原型在进化中恰好解决了该根因 → 故模仿其对应维度

痛点不是引言里硬凑出来的"写作技巧"，而是从生物原型和仿生能力**倒推**出来的逻辑锚点。先确定能仿什么、仿到什么保真度，再回头写痛点，逻辑才闭环。顺序是：先定能仿什么生物原型（哪几个维度能真仿到）→ 再倒推痛点清单 → 再写引言。

## 六、污染物选择策略

不预先固定某一种。拟先列候选清单，让 AI 针对性设计，再挑效果好的一种做实验。

**候选标准（两条同时满足）**：
1. 环境意义大（新污染物管控清单/全球关注，如 PFAS）
2. 有特殊机制（能触发有意思的仿生设计）

**清单格式（每一条必须标注）**：

| 污染物 | 环境意义 | 特殊机制 | 生物原型线索 |
|--------|---------|---------|-------------|
| （示例）PFAS | 新污染物管控清单，全球关注 | 极强 C-F 键，常规吸附剂亲和力弱 | 能代谢/耐受 PFAS 的微生物表面蛋白 |

清单作用是选"验证案例"而非"立论起点"（立论已确定用思路二共性难题）。选案例须看能否体现共性难题的解决，不只是"好不好仿生"。这样挑的不是"哪个污染物最热门"，而是"哪个污染物的特殊机制 + 生物原型线索 + 可仿维度组合最漂亮"。

## 七、AI 设计载体选择：用 Codex IDE，不用 ChatGPT 对话

此前曾用 ChatGPT 多轮对话 + 模型 A/B 对抗（A 出方案 → B 找问题 → A 复核）设计材料（无仿生约束），效果较好。但这次仿生设计约束多（多维度对应、仿生保真度、可合成性、痛点-仿生对应、引言逻辑闭环），单对话内讲清楚困难。

**判断依据（第一性原理）**：ChatGPT 对话和 Codex IDE 的本质差别不在模型能力（底层模型可以一样），而在**约束的承载位置**：

- ChatGPT 对话：约束承载在上下文窗口里，靠模型每次推理时"想起来"。对抗轮次一多，约束开始被稀释。
- Codex IDE：约束承载在文件系统里（任务书、提示词模板、对抗脚本、评判标准），模型每次推理都重新读取文件，约束不会丢。

仿生设计恰恰是约束多、约束之间有依赖、约束要在多轮对抗中保持稳定的任务，天然适合文件系统承载，不适合上下文窗口承载。

**对抗方法的保留与改造**：A 出方案 → B 攻击 → A 复核的对抗思路保留，但每轮只传修改差异而非整方案，下轮只读当前方案与未解决问题而非完整历史。这和 ADRMATS 里"每轮传修改差异非整方案"的思路一致。

## 八、BMDL 仿生设计库的使用定位（参考非强制）

### 8.1 库的基本情况

BMDL（隶属 ADRMATS）的 PostgreSQL 数据库已建成，结构完整。核心 schema 为 `bmdl`，包含：

- `biological_prototypes`：48 个生物原型，26 个字段（含 `biomimetic_dimension` 仿生维度、`mechanisms` 机制、`design_translation` 设计转化、`engineering_constraints` 工程约束、`honesty_ledger` 诚实性台账、`do_not_intrinsic` 不应移植特质、`verification_tier` 验证等级、`causal_chain` 因果链等）
- `pollutant_profiles`：44 种污染物（含 `molecular_features` 分子特征、`likely_interactions` 可能作用机制、`applicable_prototypes` 适用原型）
- `match_weights`：130 条原型×污染物匹配权重（含 `matching_basis` 匹配依据、`direct_evidence` 直接证据标志、`bound_mechanism` 绑定机制）
- `performance_data`：3015 条性能数据
- `pollutant_aliases`：216 条污染物别名

另有 `pollutant_kb`（污染物知识库 + 匹配矩阵）、`adsorption_literature`（吸附文献 + 向量索引）、`wq_constraints`（真实水质约束数据，含二级出水统计）、`bmdl_staging`（暂存区）。

### 8.2 使用原则：参考非强制，以质量为准

**BMDL 库可以作为参考，但不写入硬流程，一切以质量为准。** 此前试用发现，库虽有一定帮助，也会约束思考，出来的结果不一定会特别好。

因此：
- 设计阶段可以参考 BMDL 库的 `biological_prototypes` 寻找原型线索、参考 `match_weights` 获取匹配依据、参考 `design_translation` 和 `engineering_constraints` 获取设计转化和工程约束
- 但**不强制使用**库内的原型或匹配结果，SOTA 大模型的基础能力已足够独立设计
- 设计完成后，须进行**自我批评与自我反思**：评估用了库到底有没有变好，有没有限制思考。这是流程的必经步骤，不是可选环节

### 8.3 数据库访问约束

- 数据库地址：`pgm-uf681weh9bztsd3reo.rwlb.rds.aliyuncs.com:5432`，库名 `WQ_data`

## 九、Codex 任务结构建议

如果用 Codex 搭建，建议如下组织（供 Codex 规划时参考，非硬性规定）：

```
仿生吸附剂设计任务/
├── design_brief.md          # 任务书：约束条件、评判标准、仿生保真度要求
├── prompt_templates/         # 提示词模板（参数化，可复用）
│   ├── designer.md           # 设计者角色提示词
│   ├── attacker.md           # 攻击者角色提示词（找问题）
│   └── reviewer.md           # 复核者角色提示词
├── candidates/               # 每个候选污染物的参数文件
│   ├── PFAS.yaml
│   ├── heavy_metal_complex.yaml
│   └── ...
├── rounds/                   # 对抗轮次输出（自动归档）
│   ├── round_1_design.md
│   ├── round_1_attack.md
│   ├── round_1_review.md
│   └── ...
└── final/                    # 最终方案
    └── design_report.md
```

好处：任务书稳定只写一次；提示词模板可复用，换个污染物只改 `candidates/` 里的参数；对抗轮次自动归档可回溯；**周佳奇自己能跑**，不依赖导师每次在场。

对抗脚本的角色分工对应此前经验：designer 出方案 → attacker 找问题 → reviewer 复核裁决，每轮从文件读取当前方案和未解决问题，而不是把整个对抗历史塞进上下文。

**评判标准（至少包含）**：
- 仿生保真度：硬对应维度达标没（分级孔道/亲疏水/刺激响应/再生中至少2个硬对应）
- 可合成性：周佳奇做得出吗（实验室条件约束）
- 痛点对应：引言痛点与仿生能力是否因果链闭环
- 仿生非包装：是否落在了传统合成做不到的能力上

## 十、推进顺序

1. 列污染物候选清单（满足环境意义 + 特殊机制，标注生物原型线索）
2. （可选）只读访问 BMDL 库，从 `biological_prototypes` 和 `pollutant_profiles` 获取参考线索
3. 写仿生吸附剂设计任务书（约束条件、评判标准、仿生保真度、可合成性、痛点对应）
4. 提炼可复用提示词模板（designer/attacker/reviewer 三角色）
5. 每个候选污染物填参数，调用 SOTA 大模型 + 多轮对抗
6. 评估方案：仿生保真度 + 可合成性 + 痛点对应 + 仿生非包装
7. **自我反思**：评估用了 BMDL 库有没有变好/有没有限制思考（必经步骤）
8. 挑1个做实验

## 十一、相关背景信息（供 Codex 了解上下文）

**课题关联**：潘尧的醌基电催化剂工作中，redox-gated quinone-CO₂ interaction、脉冲电催化再生，本身就是"动态响应/可编程"仿生理念的电化学版本。如果仿生吸附材料也走"动态响应"路线，电催化和吸附两块工作可互为论据，形成体系。

**水质场景**：用纯水或传统市政污水出水（如二级出水）。BMDL 库的 `wq_constraints.adsorption_water_quality_stats` 有真实污水厂水质统计可参考（非强制）。
