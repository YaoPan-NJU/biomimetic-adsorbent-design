# 隔离角色与研究门禁协议

本项目采用 Qwen 主代理监督、隔离上下文子代理执行的工作方式。它借用多代理工作流的“有界输入—独立复核—门禁放行”原则，但不启动常驻代理、定时心跳或外部邮箱服务。Git 提交、状态更新和门禁裁决由主代理串行完成。仿生设计研究采用联网深度搜索，不依赖本地数据库。

## 权限链

潘尧是最终决策者；Qwen 主代理负责证据整合、复现实验、门禁和 Git 同步；设计者、攻击者和 Reviewer 只处理被点名的输入并写入指定文件。任何子角色都不得自行推进阶段、修改项目规约、改写他人产物或执行 Git 操作。

## 角色输入边界

| 角色 | 可读取 | 不得读取 | 唯一输出 |
|---|---|---|---|
| 深入设计者 | 入围污染物、证据边界、科学问题与工程难题定义 | 其他角色的隐藏推理、后续攻击意见 | 当前轮 `design.md` |
| 攻击者 | 当前方案、证据台账、攻击清单、创新性检查清单 | 设计者隐藏推理、未来修订稿 | 当前轮 `attack.md` |
| 裁决Reviewer | 当前方案、证据台账、攻击意见、评分表、创新性检查结果 | 完整历史链和角色身份 | 当前轮 `review.md` |

## 门禁

1. `G1 shortlist`：污染物候选、科学问题与工程难题定义、冻结问题陈述和来源边界完成。
2. `G3 concepts`：深入设计均具有静态硬对应、可证伪预测和创新性检查通过。
3. `G4 adversarial`：所有方案完成两轮，前两套完成第三轮；所有critical/high问题关闭或否决该方案。
4. `G5 delivery`：主/备方案、SOP、实验对照、失败路线、附件和引用台账完整。
5. `G6 continuity`：状态一致、密钥扫描通过、远端已推送、全新克隆复核通过。

每个门禁必须由主代理独立检查产物和校验结果，并在 `orchestration/gates/` 写入记录后，才能进入下一阶段。门禁记录只保存可审计的结论和证据，不保存模型隐藏推理。

## 失败行为

- 证据不支持：降级主张、补证或否决，禁止编造补齐。
- 单方案低于85分或仍有critical/high问题：修订或淘汰，不降低门槛。
- 多个角色修改同一文件：停止合并，由主代理从各自独立产物重新整合。
- 需要改变主场景、评分合同或最终交付边界：升级给潘尧决定。
- 创新性检查未通过：不得进入正式评分，必须修改或重新设计。
