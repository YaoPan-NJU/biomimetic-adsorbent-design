# 千方案计划运行注记（OPS_NOTES）

本文件记录工作流执行的工程教训与脚本渲染规则，供任何执行者（Claude Code 或千问 Qwen）恢复推进时直接使用，避免重复试错。

## 三种已确认的失败模式与对策

1. **引擎按 `meta.name` 缓存编译产物**：同名脚本修改后重跑，实际执行的仍是旧版本（表现为同样堆栈、同样约 70ms 内秒崩）。对策：每个脚本使用唯一 `meta.name`，建议格式 `fresh1000-pB-<slug>-<slot>-a<尝试次数>`。
2. **大返回值导致结果持久化失败**：工作流返回数百 KB（多槽 × 多轮 × 全文 Markdown）时，输出文件写成 0 字节、journal 断裂，35 分钟计算全部丢失。对策：各角色代理用 Write 工具把全文写入 `rounds/fresh_1000/<slug>/SCHEMES/_wip/<slot>_<role>_r<轮次>.md`（唯一产物），只返回瘦身的结构化摘要（分数、问题台账一句话摘要、证据条目）。
3. **会话/作业中断丢失整批**：整批 5 槽作为一个工作流时，会话中断会丢失全部在途计算。对策：单槽单工作流（15-25 分钟），槽完成即提交推送；中断后从 `STATUS.yaml` 的逐槽状态与 `_wip/` 残留文件恢复，未完成的槽整体重跑（`_wip/` 会被覆盖，安全）。

## 脚本渲染规则

- 工作流脚本不能用文件系统 API，但代理可以读写仓库文件；参数**不要依赖 `args`**（传递形态不稳定），把批次数据作为 JS 字面量直接嵌入脚本（`const A = {...}`）。
- 用 `parallel()` 不要用 `pipeline()`（后者在本环境表现不稳定）。单槽脚本直接 `await processSlot(A.slot)`。
- 角色提示词保持完整质量标准（9 项闭环、攻击七维、评分量表、创新性五维），只改输出路由。
- 攻击者/裁决者通过 Read 工具读 `_wip/` 文件获得上游产物（有界输入），不在提示词中嵌入长文本。

## 单槽执行循环（监督者操作）

1. 渲染单槽脚本（唯一 meta.name，嵌入该槽定义 + 已知陷阱 + 仓库历史教训）；
2. 启动工作流，等待完成通知；
3. 从 `_wip/<slot>_design_r<N>.md`、`_attack_r<N>.md`、`_review_r<N>.md` 组装正式方案文件 `SCHEMES/S<序号>_<slot>_<题名简称>.md`（含状态、得分、轮次、重建建议）；
4. 去重后把证据条目追加进 `research/evidence/evidence_ledger.csv`；
5. 单独提交该方案并推送 `origin/Qwen`；
6. 更新 `STATUS.yaml` 逐槽状态（passed / needs_rebuild / terminated 与尝试次数）；
7. passed → 下一槽；needs_rebuild 且尝试次数 <3 → 换新角度或带问题台账重渲染重跑；尝试 3 次仍不过 → terminated，记入耗尽条款。

## 恢复判据

- **通知报 failed/秒崩也可能是误投递**（2026-07-21 四起实证）：重跑前先数 journal 条目与代理转录数、看输出文件大小；多小时级长任务"秒崩"几乎必为误投递，实际常在运行中。禁止据失败通知直接重跑同名脚本。
- **同一任务可能重复投递相互矛盾的载荷**（2026-07-21 A03 实证：77 分与 74 分两版）：权威次序为非零输出文件 ≥ journal（需多次读取确认，条目可闪现后消失）＞ 通知；与已持久化产物（设计全文等）互洽的版本为真。通知截断点之后的正文不可凭记忆补写——重建必须标注来源，补写即造假。发现矛盾时：停掉基于可疑载荷已启动的下游工作，按权威版本重判。
- **同一轮的通知 design 与 journal design 可能异版**（2026-07-21 A02 r4 实证）：向下一轮嵌入上一轮方案时一律取 journal 版本（该轮真实产物），台账如取自通知须标注 provenance 并接受下一轮裁决者对实际嵌入版本的再评估。
- 工作流"完成"但输出 0 字节：先查 `_wip/` 是否有该槽产物、journal 是否有 result 条目；两者皆无则按未完成重跑该槽。
- 子代理转录只有 user 提示、无 assistant 输出：该代理夭折（瞬时 API 故障），用 `resumeFromRunId` 恢复或整体重跑。
- **任务通知误投递**（2026-07-21 事故）：同一任务可能先送达一份陈旧或捏造的 result 载荷（如 A02/A05 先收到 79/84 分载荷，journal 权威值实为 65/54 分，问题台账完全不同）。规则：**任何通知载荷在据其行动前必须与 journal 的 result 条目交叉核验**（journal 写入可能延迟数分钟，需重查）；journal 缺失时核对载荷内部一致性（评分细目之和等于总分、问题台账与裁决结论互洽），并等待 journal 落盘。误投递载荷的特征：分数偏高、问题数偏少、缺乏裁决者独立再核验痕迹。
- journal 的 result 条目是**代理原始结构化返回**（如 REVIEW 对象本身），不是工作流 `return` 的外层包装；提取时不要套 `["review"]`。
